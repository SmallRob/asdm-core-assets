# Retrospective Specifications

## Purpose
Defines the format, process, and best practices for conducting effective sprint retrospectives.

## Retrospective Principles

### Prime Directive
"Regardless of what we discover, we understand and truly believe that everyone did the best job they could, given what they knew at the time, their skills and abilities, the resources available, and the situation at hand."

### Core Values
- **Safe**: No blame or judgment
- **Honest**: Open and transparent communication
- **Actionable**: Focus on improvements
- **Time-boxed**: Respect time limits

## Retrospective Formats

### Start-Stop-Continue
Best for: General improvement

**Start**: What should we start doing?  
**Stop**: What should we stop doing?  
**Continue**: What should we continue doing?

### 4Ls (Liked, Learned, Lacked, Longed For)
Best for: Learning-focused teams

**Liked**: What did you enjoy?  
**Learned**: What new knowledge did you gain?  
**Lacked**: What was missing?  
**Longed For**: What did you wish for?

### Mad-Sad-Glad
Best for: Emotional check-in

**Mad**: What frustrated you?  
**Sad**: What disappointed you?  
**Glad**: What made you happy?

### DAH (Drop, Add, Keep, Improve)
Best for: Process improvement

**Drop**: What should we eliminate?  
**Add**: What should we introduce?  
**Keep**: What should we maintain?  
**Improve**: What should we enhance?

### Fishbone (Ishikawa Diagram)
Best for: Root cause analysis

Categories:
- People
- Process
- Technology
- Environment

### Timeline
Best for: Detailed sprint analysis

1. Create timeline of sprint events
2. Mark highs and lows
3. Discuss patterns
4. Identify improvements

## Retrospective Process

### Phase 1: Set the Stage (5-10 min)
- Welcome participants
- Review retrospective purpose
- Establish safety (safety check)
- Set time box

### Phase 2: Gather Data (15-20 min)
- Collect facts and feelings
- Use selected format
- Encourage all voices
- No discussion yet

### Phase 3: Generate Insights (20-30 min)
- Discuss collected items
- Identify patterns
- Find root causes
- Group similar items

### Phase 4: Decide What to Do (20-30 min)
- Prioritize issues
- Brainstorm solutions
- Select action items
- Assign owners and due dates

### Phase 5: Close (5-10 min)
- Summarize action items
- Appreciate contributions
- Gather feedback on retrospective
- End on positive note

## Facilitation Guidelines

### Pre-Retrospective
- [ ] Schedule meeting
- [ ] Prepare materials
- [ ] Gather sprint metrics
- [ ] Review previous action items
- [ ] Select appropriate format

### During Retrospective
- [ ] Create safe environment
- [ ] Ensure equal participation
- [ ] Time-box effectively
- [ ] Focus on actionable items
- [ ] Document everything

### Post-Retrospective
- [ ] Share summary with team
- [ ] Track action items
- [ ] Update team processes
- [ ] Schedule follow-up

## Action Item Guidelines

### SMART Criteria
- **Specific**: Clear and well-defined
- **Measurable**: Can determine completion
- **Assignable**: Has a clear owner
- **Realistic**: Achievable within timeframe
- **Time-bound**: Has a due date

### Example Action Items
✅ "Implement PR review checklist (Owner: Dev 1, Due: Feb 5)"
✅ "Reduce standup time to 15 min (Owner: Scrum Master, Due: Next standup)"
✅ "Update deployment documentation (Owner: Dev 2, Due: Feb 10)"

❌ "Improve code quality" (too vague)
❌ "Be better at testing" (no owner, no due date)

## Retrospective Metrics

### Sprint Metrics to Include
- Committed vs. completed points
- Velocity trend
- Bug count
- Cycle time
- Team satisfaction

### Retrospective Effectiveness
- Action item completion rate
- Recurring issues trend
- Team engagement score

## Safety Check

### Technique
Ask team to rate safety (1-5) anonymously:
- 1: Not safe to speak
- 2: Hesitant to share
- 3: Neutral
- 4: Comfortable sharing
- 5: Very safe

### If Average < 3
- Address safety concerns
- Consider anonymous input
- Review team dynamics
- Escalate if necessary

## Common Anti-Patterns

### Blame Game
❌ Focusing on who made mistakes
✅ Focus on what can be improved

### Complaint Session
❌ Venting without action items
✅ Convert complaints to improvements

### No Follow-Through
❌ Action items not tracked
✅ Assign owners and track completion

### Same Issues Every Retro
❌ Not addressing root causes
✅ Dig deeper, implement solutions

### Dominated Discussion
❌ One or two people doing all talking
✅ Use facilitation techniques to ensure participation

## Retrospective Output Format

```markdown
# Sprint X Retrospective

**Date**: YYYY-MM-DD  
**Sprint**: Sprint X  
**Facilitator**: [Name]  
**Participants**: [List]

## Sprint Metrics
- Committed: X points
- Completed: Y points
- Velocity: Z points
- Completion Rate: Y/X%

## What Went Well
- [Item 1]
- [Item 2]

## What Could Be Improved
- [Item 1]
- [Item 2]

## Action Items
| Item | Owner | Due Date | Status |
|------|-------|----------|--------|
| Action 1 | Owner 1 | Date 1 | Open |
| Action 2 | Owner 2 | Date 2 | Open |

## Follow-up from Last Retro
| Item | Status | Notes |
|------|--------|-------|
| Previous Action 1 | Done | Successfully implemented |

## Team Health Check
- Safety Score: X/5
- Satisfaction: X/5
```

## Remote Retrospective Tips

### Tools
- Miro/Mural: Collaborative boards
- Menti/Slido: Anonymous polling
- Zoom/Teams: Video conferencing
- Trello/Jira: Action item tracking

### Best Practices
- Use video (cameras on)
- Utilize breakout rooms for small groups
- Use collaborative whiteboards
- Have a co-facilitator for technical issues
- Allow extra time for technology setup

## Frequency

### Standard
- End of each sprint

### Options
- Mid-sprint check-in (quick pulse)
- Project retrospective (after release)
- Team health check (monthly)
